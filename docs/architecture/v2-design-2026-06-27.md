## 0. 一句话目标

把 **3 个 1100+ 行的"步骤流怪物"** 拆成 **Action → Step → Workflow → Engine** 4 层架构，让 **新流程开发从"改 1000 行单文件"变成"组装 10 个 50 行的 step"**，**新机型适配从"埋点硬编码坐标"变成"写一条适配规则"**。

| `native/index.ts` | **1635** | 原生模块桥接 + 便捷函数 + 类型定义混在一起 | **改 native API 要小心不要碰错类型** |
| `android/.../AutomationModule.kt` | **1834** | AccessibilityService 全部方法 + 弹窗 + 震动 + 截屏 + 多窗口 | **Kotlin 端同样混乱** |

**对比业内标准**：React Native 单文件 500 行就要警惕，**2000 行就是"必须重构"红灯**。

### 1.2 关注点未分离（架构性痛点）

BaoliService.ts 第 246-852 行（举例）：**一个步骤函数里同时干 5 件事**：

```typescript
// ========== 步骤3：点击"工作台" ==========
logToBoth('info', '[步骤3] 点击"工作台"...');              // ① 日志
const workbenchNode = await findNode('工作台');              // ② UI 查找
if (workbenchNode) {                                         // ③ 业务判断
  logToBoth('success', '找到 @ (...)');
  await humanTap(workbenchNode.x, workbenchNode.y);          // ④ UI 操作
} else {
  await waitForUserGo('找不到"工作台"', '请手动找...');       // ⑤ 异常恢复
  // 用户 GO 后？
}
```

**5 个关注点缠在一起** → 测试要 mock 一切 → 没法独立单测步骤 → bug 改了 A 步骤，B 步骤坏了。

### 1.3 业务流重复（千机 vs 保利）

千机 4 步 + 保利 17 步 = 21 步里**至少 8 步是同一模式**：

| 步骤 | 千机版本 | 保利版本 | 重复度 |
|---|---|---|---|
| 开 APP | `openApp('qianji')` | `openApp('wework')` | 100% |
| 找节点 + tap | 同 | 同 | 100% |
| 找节点 + retry | 同 | 同 | 100% |
| GO 恢复 | 同 | 同 | 100% |
| 截屏 + 标注 | 同 | 同 | 100% |
| 写库 | 同 | 同 | 100% |
| 拟人化 tap/swipe | 同 | 同 | 100% |
| 长按触发粘贴弹窗 | 同 | 同 | 100% |

**8 处重复**，改一处要改 8 处。

### 1.4 机型分支硬编码（v2 老板拍 D 方案）

`deviceModel.ts` 已做机型分支，但**调用方 100+ 处散落**：

```typescript
// 每个步骤都要这样写：
const coord = await getPasteMenuCoord();
await zbbAutomation.tap(coord.x, coord.y);
```

**应该封装成 Action**，调用方不关心机型。

### 1.5 异常处理分散（GO 机制痛点）

v1.6.4 落地 GO 机制，**9 处接入都长一样**：

```typescript
const node = await findNode('XXX');
if (!node) {
  await waitForUserGo('[步骤3] 找不到"XXX"', '请手动找...');
  // 然后呢？
}
```

**9 处都"GO 后怎么继续"由调用方决定** → 实际多数是直接继续 → **应该封装成 "GO 后自动重试"**。

### 1.6 测试覆盖率（实测量）

老板 6 月 23-26 日踩坑数据：
- **GO 震动 30s bug** = commit 799bd4c（vibration 永久循环）→ 修在 cda4072
- **getAllTextNodes 5 缺陷** = 老板"查不到"反馈 → 修在 0ffe0c0
- **vivo 机型匹配失败** = 老板 vivo 走华为坐标 → 修在 ea9be2a

**核心问题：单测覆盖率约 0%，所有 bug 都是用户/老板真机踩出来**。架构不重构 → 单测永远写不出来。

## 3. 核心设计原则（7 条铁律）

> **每条都对应一个具体痛点，违反一条 = 引入新的技术债。**

### P1: Action 原子化（解决 1100 行怪物）

**每个 Action = 一个动词 + 入参 + 出参，不超过 50 行**：

```typescript
// ✅ 正确：原子 Action
async function tap(target: NodeTarget): Promise<void>     // 不超过 30 行
async function findText(text: string): Promise<Element | null>  // 不超过 50 行
async function longPress(target: NodeTarget, ms: number): Promise<void>

// ❌ 错误：上帝函数
async function step3_clickWorkbench() { ... }  // 50 行 + UI + 业务 + 异常
```

### P2: Step = Action 编排，不带业务判断

**Step 只负责"按顺序调 Action"，业务判断（节点是否找到 → GO 还是继续）由 Workflow 决定**：

```typescript
// ✅ Step = 编排（无业务判断）
const step = pipe(
  findText('工作台'),          // Action 1
  tapAt('工作台'),              // Action 2（自动取 findText 结果）
  delay(2000),                  // Action 3
);

// ✅ Workflow = 业务判断（GO 决策）
if (!result) {
  await waitForUserGo('找不到"工作台"', '...');
  return retryStep(step);       // 明确"GO 后重试"
}
```

### P3: Adapter 抽象机型（解决 D 方案痛点）

**所有"按机型变化"的东西走 Adapter，调用方零感知**：

```typescript
// ❌ 错误：调用方处理机型
const coord = await getPasteMenuCoord();
await zbbAutomation.tap(coord.x, coord.y);

// ✅ 正确：Action 内部处理
await tapText('工作台');   // 内部自动按机型取坐标
```

### P4: 状态机显式化（解决 21 步骤状态混乱）

**Workflow 状态 = 显式 enum + 转换函数，不靠隐式 await 顺序**：

```typescript
type State = 'idle' | 'openingApp' | 'findingNode' | 'tapping' | 'goPause' | 'completed' | 'failed';

// ✅ 转换函数明确
function transition(state: State, event: Event): State { ... }
```

### P5: GO 机制升级到"自动重试"

**v1.6.4 GO 后由调用方决定怎么继续 → v2 GO 后默认自动重试，调用方可覆盖**：

```typescript
// v2 默认行为
await tapTextWithRetry('工作台', { maxRetries: 3, goOnFail: true });

// 等价于：失败 → GO → 用户点 GO → 自动重试
```

### P6: 数据模型集中（解决剪贴板桥接散落）

**所有"读/写数据库"走 data/ 层，业务代码不直接调 SQL**：

```typescript
// ❌ 错误：业务代码直接 SQL
await DatabaseService.insertBaoliReport(...);

// ✅ 正确：业务代码调 data 层
await reports.insertBaoli(customer);   // data/reports.ts 封装
```

### P7: TypeScript 严格模式（解决 native 桥接类型混乱）

**tsconfig 开 strict + noUncheckedIndexedAccess + exactOptionalPropertyTypes**：

```typescript
// 强类型，禁止 any
function tap(target: NodeTarget): Promise<void>    // target 必须有 bounds
```

## 5. 核心抽象详解（流程结构 + 4 层架构）

### 5.1 流程结构（端到端总览）

> **v2 业务核心 = 持续接龙循环**：每客户走 千机 → 保利（两轮） → 回千机 接下一个，直到用户手动停止或无可报备客户。
>
> 本节基于 `references/qianji-flow-architecture-2026-06-22.md` + `references/baoli-flow-architecture-2026-06-22.md` + `references/zbb-relay-loop-architecture.md` 三份最新流程图（commit `22b6ac7`）。**5.2-5.5 = 4 层架构 = Action/Step/Workflow/Engine**。

#### 5.1.1 完整链路（一组客户 = 1 个循环周期）

```
[用户点"测试千机端"]
   ↓
千机端 startQianjiFlow()               ← 首次入口（首页按钮）
   ├─ step1: 拉起千机（launchApp + maybePause + Gamma delay）
   ├─ step2: 读原生树 + 待报备预检
   │        └─ 0 待报备 → lastExitReason='no_pending' + pressHome + return（闸门挡 step3+4）
   ├─ step3: 转发流程（4 子步：找"报备审核" → 判断含"保利" → 找"转发" → "复制" → 解析客户信息）
   │        ├─ 没保利 → Toast + 震动 + lastExitReason='no_baoli' + return（闸门挡 step4）
   │        └─ 不是保利项目 → 留在千机界面（不 pressHome，让用户手动处理）
   └─ step4: 移交保利（maybePause + pGammaDelay + baoli.execute()）
                              ↓
保利端 execute()                        ← 入口（千机 step4 / 首页按钮 / 自身递归）
   ├─ 第一阶段导航（步骤 1-7）
   │   pressHome → 企业微信 → 工作台 → 云和家 → 项目 → 报备表单
   ├─ 第二阶段填表（步骤 8-17，fillForm() 内）
   │   触发 EMUI/OriginOS 粘贴弹窗 → getAllTextNodes → 9 字段正则解析 →
   │   选分期 → 选项目 → 确认 → 智能识别 → 报备 → 等报备完成
   └─ detectResult(1) 分支
       ├─ 情况 1：疑似重号 → 震动 + Toast → 等 30s 用户点取消 → todayBaoliCount++ + killZbbProcess
       └─ 情况 2：报备成功 → 上滑 + 找"上传附件" + 用户手动截图确认（GO 圆点）+ pressBack
              ↓
       handleSecondRound()             ← 第一轮成功后
       ├─ 步骤 1 + fillForm('郑州市三村杓袁7号地项目-保利山水和颂') + detectResult(2)
       └─ detectResult(2) 情况 2 → handleSuccessCase(2)
              ↓
       步骤 19-20：pressBack + pressHome + launchAppWithAmStart(千机) + 找"报备有效"
              ↓
       震动 + Toast("✅ 已完成报备") + showScreenshotButton (GO 圆点) + 等用户点 GO
              ↓
       exitMiniProgram×2（多任务键 + 垃圾箱）
              ↓
接龙：testOnlyQianjiFlow()             ← 千机 step1+2+3（不调 step4，避免死循环）
   ├─ 'no_pending' → Toast + return（结束）
   ├─ 'no_baoli'   → Toast + return（结束）
   └─ 'has_customer' → this.isRunning = false 释放锁 → this.execute() 递归
              ↓
       handleSuccessCase(2) → ...（又到循环点）
```

#### 5.1.2 千机端 4 步（详细）

| 步骤 | 方法 | 行为 | 退出条件 |
|---|---|---|---|
| 1 | `stepOpenQianji()` | `launchAppWithAmStart('com.lianjia.anchang', 'MainActivity')` + Gamma delay + `maybePause`（8% Poisson 停顿） | 失败 → 抛错（不上抛 lastExitReason） |
| 2 | `stepRecognizeInterface()` | Gamma delay 2-3s + `getAllTextNodes()` 存 `this.lastTextNodes` + 待报备预检（最多 3 次下拉刷新） | `lastExitReason='no_pending'` → startQianjiFlow L587 闸门挡 step3+4 |
| 3 | `stepFindAndCollectCustomer()` | 找"报备审核" → 界面必须含"保利" → 找"转发"（列表页） → 找"转发"（联系人页 Y 最大） → 找严格 `text==='复制'` → `assembleKeyValueLines + parseClipboardText` 解析客户信息 | `lastExitReason='no_baoli'` → startQianjiFlow L597 闸门挡 step4（06-21 修：3-1/3-2/3-3 早退漏设 lastExitReason，**v2 必修**） |
| 4 | `stepJumpToReportApp()` | `maybePause` + `pGammaDelay(500,1500)` + `baoli.execute()`（**无参！保利端自己从原生树读**） | 千机 step3 退出 → 跳过 |

**千机 2 个入口**：
- `startQianjiFlow()`（首页"测试千机端"按钮）：step1+2+3+4，**入口处重置 `lastExitReason = null`**（L577）
- `testOnlyQianjiFlow()`（保利成功后接龙）：step1+2+3（**不调 step4**），**入口处重置 `lastExitReason = null`**（L626）

**`lastExitReason` 三态**：
- `null` = 继续
- `'no_pending'` = 当前无待报备客户（已 Toast + `pressHome`）
- `'no_baoli'` = 当前没有保利客户（已 Toast + 震动 + **不回桌面**，让用户手动处理）

#### 5.1.3 保利端 17 步（详细，按真实业务顺序）

**⚠️ 代码里 `logToBoth` 步骤编号错乱**（代码标 [步骤2] 实际是真实步骤 3 等）。下面按**真实业务顺序 1-17**：

| 真实步骤 | 代码 log 标号 | 行为 | 关键点 / 兜底 |
|---|---|---|---|

### 5.2 Action 层（原子动作）

**Action 签名约定**：

```typescript

// 出参：明确的结果（成功 / 失败 / 数据）
// 副作用：通过 zbbAutomation（native）唯一出口
// 错误：throw typed error，不静默失败

type NodeTarget = 
  | { kind: 'text'; text: string }
  | { kind: 'coord'; x: number; y: number; dp?: boolean }
  | { kind: 'desc'; desc: string };

interface ActionResult<T = void> {
  ok: boolean;
  data?: T;
  error?: ActionError;
}

async function tap(target: NodeTarget, opts?: { jitter?: boolean }): Promise<ActionResult> {
  // 内部自动：机型分支 + dp 归一化 + 拟人化 + retry

}
```

```typescript
// engine/executor.ts
export async function runWorkflow(
  workflow: Workflow,
  context: WorkflowContext
): Promise<WorkflowResult> {
  let state = workflow.initialState;
  
  for (const s of workflow.steps) {
    state = transition(state, 'stepStart', s.name);
    
    try {
      const result = await executeStep(s, context);
      if (!result.ok && s.goOnFail) {
        await waitForUserGo(s.goOnFail.reason, s.goOnFail.hint);
        if (s.goOnFail.action === 'retry') {
          const retryResult = await executeStep(s, context);
          if (!retryResult.ok) throw new ActionError(s.name, retryResult.error);
        }
      }
      state = transition(state, 'stepEnd', s.name);
    } catch (e) {
      state = transition(state, 'stepFail', s.name);
      throw e;
    }
  }
  
  return { ok: true, state };
}
```

### 6.2 engine/go.ts（GO 机制升级）

```typescript
// engine/go.ts
export interface GoOnFailConfig {
  reason: string;          // 用户看到的第一行
  hint: string;            // 用户看到的第二行
  action?: 'retry' | 'continue' | 'abort';  // 默认 retry
  maxRetries?: number;     // 默认 3
}

export async function withGoOnFail<T>(
  fn: () => Promise<T>,
  config: GoOnFailConfig
): Promise<T> {
  try {
    return await fn();
  } catch (e) {
    await waitForUserGo(config.reason, config.hint);
    if (config.action === 'abort') throw e;
    
    const maxRetries = config.maxRetries ?? 3;
    for (let i = 0; i < maxRetries; i++) {
      try {
        return await fn();
      } catch (e2) {
        if (i === maxRetries - 1) throw e2;
      }
    }
    throw e;
  }
}
```

## 7. 数据流（State / Event / Data）

### 7.1 状态机显式化（解决 21 步骤状态混乱）

```typescript
// engine/state.ts
export type WorkflowState =
  | { kind: 'idle' }
  | { kind: 'running'; currentStep: string; startedAt: number }
  | { kind: 'goPause'; step: string; reason: string; since: number }
  | { kind: 'completed'; totalDuration: number }
  | { kind: 'failed'; step: string; error: string };

export type WorkflowEvent =
  | { kind: 'stepStart'; step: string }
  | { kind: 'stepEnd'; step: string; ok: boolean }
  | { kind: 'goRequested'; step: string; reason: string }
  | { kind: 'goConfirmed'; step: string }
  | { kind: 'workflowAbort' };

export function transition(state: WorkflowState, event: WorkflowEvent): WorkflowState {
  switch (state.kind) {
    case 'idle':
      if (event.kind === 'stepStart') return { kind: 'running', currentStep: event.step, startedAt: Date.now() };
      break;
    case 'running':
      if (event.kind === 'goRequested') return { kind: 'goPause', step: event.step, reason: event.reason, since: Date.now() };
      if (event.kind === 'stepEnd' && event.ok) return state;
      if (event.kind === 'stepEnd' && !event.ok) return { kind: 'failed', step: state.currentStep, error: 'step failed' };
      break;
    case 'goPause':
      if (event.kind === 'goConfirmed') return { kind: 'running', currentStep: state.step, startedAt: Date.now() };
      break;
    // ...
  }
  return state;  // 无效转换保持
}
```

**优势**：状态转换明确 → 单测能覆盖每个分支 → bug 定位精确到"哪个转换错了"。

## 附录 B：术语表

| 术语 | 含义 |
|---|---|
| **Action** | 原子动作（如 tap / findText / longPress），单个文件 ≤ 50 行 |
| **Step** | 一组 Action 的有序编排（用 pipe / sequence 表达） |
| **Workflow** | 业务流（千机 4 步 / 保利 17 步 / 接龙循环） |
| **Engine** | Workflow 执行器（状态机 + retry + GO + log） |
| **Adapter** | 设备机型分支封装（vivo vs nova 7 5G） |
| **GO** | 通用"暂停 + 用户确认 + 继续"机制（utils/goResume） |
| **dp 归一化** | 用 dp 而非 px 表达坐标，跨屏宽保持相对位置 |
| **接龙** | 老板业务核心 = 千机→保利→千机循环，处理下一个客户 |
| **新架构 / v2** | 本文档设计的目标架构 |
| **老架构 / v1** | 当前 release/user-v1.6.4 的架构（怪物文件） |
